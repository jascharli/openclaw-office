## 1. 项目概述

### 1.1 项目简介
**龙虾办公室**是一个 AI 团队工作空间监控系统，提供上帝视角的实时监控和干预能力。系统通过实时采集和分析 OpenClaw 平台的 Agent 状态、任务执行情况和大模型请求数据，为管理者提供全面的 AI 团队运行状况视图。

### 1.2 核心价值
- 📊 **实时监控**：实时查看 Agent 状态（休闲/对话/办公）和任务执行情况
- 📋 **任务管理**：任务清单、完成情况统计和任务移交功能
- 📈 **数据统计**：大模型请求数据统计、Token 使用统计和分模型分析
- 🔔 **督促功能**：及时干预 Agent 工作，确保任务按时完成
- 📤 **任务移交**：支持 Agent 间任务转交，优化工作分配

### 1.3 目标用户
- **AI 团队管理者**：需要全局监控团队运行状况
- **系统运维人员**：需要监控系统资源使用情况
- **Agent 开发者**：需要了解 Agent 性能和 Token 消耗

#### 2.2.1 Cron 任务同步与状态判断

##### 2.2.1.1 任务同步完整流程

**流程概述**：从 OpenClaw 获取 Cron 任务，解析任务信息，拆分周期性任务，判断任务状态，最后同步到本地数据库。

**详细流程**：

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│ 1. 获取任务列表 │────>│ 2. 解析任务信息 │────>│ 3. 拆分周期性任务 │
└─────────────────┘     └─────────────────┘     └─────────────────┘
          │                       │                       │
          ▼                       ▼                       ▼
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│ 4. 判断任务状态 │────>│ 5. 同步到数据库 │────>│ 6. 前端展示任务 │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

**步骤说明**：

1. **获取任务列表**
   - 执行命令：`openclaw cron list`
   - 解析命令输出，提取任务信息
   - 处理权限问题：设置正确的环境变量（HOME、USER）

2. **解析任务信息**
   - 提取字段：任务ID、名称、归属Agent、调度规则、下次执行时间、上次执行时间、状态
   - 解析调度规则，判断任务类型（每日、每小时、每X小时）

3. **拆分周期性任务**
   - 对于每小时任务：拆分为24个任务实例（0-23点）
   - 对于每X小时任务：拆分为多个整点时刻的任务实例
   - 为每个拆分的任务设置目标执行时间（target_datetime）

4. **判断任务状态**
   - 第一步：判断是否是今天的任务
   - 第二步：根据时间和状态信息判断任务执行状态
   - 处理特殊情况：暂停的任务、失败的任务、新任务

5. **同步到数据库**
   - 使用任务的计划执行时间作为 `created_at`
   - 对于已存在的任务：更新状态和信息
   - 对于新任务：创建新记录

6. **前端展示任务**
   - 查询今日创建的任务（00:00至今）
   - 过滤任务：只显示相关任务
   - 按Agent分组，计算完成情况

##### 2.2.1.2 任务创建时间（created_at）的处理

**问题描述**：同步 Cron 任务时使用当前时间作为 `created_at`，导致时区转换错误。

**错误流程**：
```
当前时间（北京）: 2026-04-01 08:xx
转换后 UTC: 2026-04-01 00:xx
存入数据库: 2026-03-31 16:xx（错误！）
```

**修复方案**：使用任务的计划执行时间作为 `created_at`

```python
# 计算任务的计划执行时间（用于 created_at）
if 'target_datetime' in task:
    # 拆分后的任务（如每2小时），使用目标时间
    task_created_at = task['target_datetime'].astimezone(UTC).replace(tzinfo=None)
elif next_info['datetime'] is not None and next_info['is_today']:
    # 有计划执行时间且是今天
    task_created_at = next_info['datetime'].astimezone(UTC).replace(tzinfo=None)
else:
    # 没有计划时间，使用当前 UTC 时间
    task_created_at = datetime.utcnow()
```

##### 2.2.1.3 任务状态判断逻辑

**状态映射**：
| 状态 | 说明 |
|------|------|
| completed | 已完成 |
| pending | 待执行 |
| blocked | 执行失败 |
| paused | 暂停 |

**判断流程**：

**第一步：判断是否是今天的任务**

| 字段 | 判断逻辑 |
|------|----------|
| Next 有值 | 判断 Next 日期是否等于今天 |
| Next 无值，用 Last | 判断 Last 日期是否等于今天 |
| Next/Last 都无值 | 视为今天的任务（新建任务） |
| Status = 'idle' | 视为今天的任务（暂停的任务） |
| Status = 'error' | 视为今天的任务（失败的任务） |

**第二步：判断任务执行状态**

| 条件 | 状态结果 |
|------|----------|
| Next 已过执行时间 + cron_status = 'ok' | completed |
| Next 已过执行时间 + cron_status = 'error' | blocked |
| Next 已过执行时间 + 其他 status | pending |
| Next 未过执行时间 | pending |
| 今日执行过 + cron_status = 'ok' | completed |
| 今日执行过 + cron_status = 'error' | blocked |
| 超过24小时未执行 | pending |
| cron_status = 'idle' | paused |
| cron_status = 'error' | blocked |

##### 2.2.1.4 周期性任务拆分规则

对于每小时/X 小时的周期性任务，需要拆分为多个任务实例显示：

| 任务类型 | 拆分规则 | 示例 |
|----------|----------|------|
| 每小时 | 拆分为24个任务，每个小时一个实例 | "每 1 小时" → 0点、1点、2点...23点 |
| 每 X 小时 | 拆分为多个任务，只显示整点时刻 | "每 2 小时" → 0点、2点、4点...22点 |

**显示规则**：
- 只在整点时刻显示（如每 2 小时：0 点、2 点、4 点...）
- 非整点时刻不显示该任务

##### 2.2.1.5 任务展示逻辑

**展示流程**：

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│ 1. 查询今日任务 │────>│ 2. 过滤任务     │────>│ 3. 分组和计算   │
└─────────────────┘     └─────────────────┘     └─────────────────┘
          │                       │                       │
          ▼                       ▼                       ▼
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│ 4. 格式化数据   │────>│ 5. 返回结果     │────>│ 6. 前端展示     │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

**过滤规则**：
1. 不显示每周 X 的任务（除非今日是周 X）
2. 不显示每月 X 日的任务（除非今日是 X 日）
3. 不显示 1 小时前已过的每日任务

##### 2.2.1.6 关键字段说明

| 字段 | 来源 | 说明 | 计算规则 | 展示价值 |
|------|------|------|----------|----------|
| Next | OpenClaw Cron 任务 | 下次执行时间 | 从命令输出解析 | 用于判断任务执行时间和状态 |
| Last | OpenClaw Cron 任务 | 上次执行时间 | 从命令输出解析 | 用于判断任务是否已执行和状态 |
| Schedule | OpenClaw Cron 任务 | 执行计划 | 从命令输出解析 | 用于判断任务类型和拆分规则 |
| Status | OpenClaw Cron 任务 | 执行状态 | 从命令输出解析 | 用于判断任务执行状态 |
| created_at | 本地数据库 | 任务记录创建时间 | 使用任务计划执行时间 | 用于判断是否是今日任务 |
| target_datetime | 拆分后的任务 | 具体的计划执行时间 | 根据拆分规则计算 | 用于设置 created_at 和判断任务状态 |
| task_id | 本地数据库 | 任务唯一标识 | "cron-" + 原始任务ID | 用于任务识别和更新 |
| agent_id | 本地数据库 | 执行Agent | 从任务信息提取，默认为'main' | 用于任务分组和展示 |
| task_status | 本地数据库 | 任务状态 | 根据判断逻辑计算 | 用于任务状态展示和统计 |

#### 2.2.2 Agent 状态判定与对话时长计算

##### 2.2.2.1 三区域状态判定逻辑

Agent 状态分为三个区域：**对话区（conversing）**、**办公区（working）**、**休闲区（idle）**

**判定优先级**（从上到下，满足即返回）：

| 优先级 | 状态 | 判定条件 |
|--------|------|----------|
| 1 | 对话区（conversing） | 有用户消息 + 有助手回复 + 最近 20 分钟内有活动 |
| 2 | 办公区（working） | 满足任一条件：<br>- subagent 正在执行<br>- cron 任务正在执行<br>- 正在调用工具<br>- 有未完成的任务上下文（最近 30 分钟内） |
| 3 | 休闲区（idle） | 以上都不满足 |

##### 2.2.2.2 对话时长计算逻辑

**数据来源**：session 文件中的消息时间戳（timestamp 字段）

```json
{
  "type": "message",
  "timestamp": "2026-03-31T20:35:16.386Z",
  "message": { "role": "user/assistant", "content": [...] }
}
```

**计算方式**：

| 状态 | 时长名称 | 计算公式 | 最大值 |
|------|----------|----------|--------|
| 对话区（conversing） | 对话时长 | 当前时间 - 最后一条用户消息时间 | 无上限 |
| 办公区（working） | 未活动时间 | 当前时间 - session 文件最后修改时间 | 24 小时 |
| 休闲区（idle） | 未活动时间 | 当前时间 - session 文件最后修改时间 | 24 小时 |

**实现逻辑**：

```python
# 对话区：使用最后一条用户消息时间计算对话时长
if status == 'conversing' and session_analysis.get('last_user_message_time'):
    conversation_duration = int((now - last_user_message_time).total_seconds())
    elapsed_time = max(0, conversation_duration)
else:
    # 其他状态：使用文件最后修改时间，最大 24 小时
    elapsed_time = min(active_minutes * 60, 86400)
```

**字段说明**：

| 字段 | 来源 | 说明 |
|------|------|------|
| timestamp | session 文件每条消息 | UTC 时间戳，格式：`2026-03-31T20:35:16.386Z` |
| first_user_message_time | 第一条 user 消息的时间 | 首次对话开始时间 |
| last_user_message_time | 最后一条 user 消息的时间 | 用于计算对话时长 |
| last_activity | session 文件最后修改时间（mtime） | 用于计算未活动时间 |

##### 2.2.2.3 状态与时长的一致性

- 对话区状态判定时间阈值：20 分钟
- 对话时长计算：无上限（真实反映对话持续时间）
- 办公区/休闲区时长：最大 24 小时

### 2.5 服务健康检测与自动恢复

#### 2.5.1 需求背景

在实际运行中，龙虾办公室依赖多个服务组件：
- **后端服务**（端口8000）：提供API服务
- **前端服务**（端口5173）：提供Web界面
- **OpenClaw Gateway**：提供Agent管理和数据对接

当OpenClaw Gateway重启后，龙虾办公室可能出现对接失败、服务不可用等问题，需要一种自动检测和恢复机制。

#### 2.5.2 功能设计

##### 2.5.2.1 检测对象与指标

| 服务 | 检测方式 | 判定标准 | 异常处理 |
|------|----------|----------|----------|
| **后端服务** | HTTP GET /health | 返回200且响应正常 | 重启后端进程 |
| **前端服务** | HTTP GET http://localhost:5173 | 返回200 | 重启前端进程 |
| **OpenClaw Gateway** | subprocess: openclaw gateway health | 退出码为0 | 执行 openclaw gateway restart |
| **OpenClaw Cron对接** | subprocess: openclaw cron list | 成功获取任务列表 | 记录警告，不触发重启 |

##### 2.5.2.2 检测策略

| 参数 | 默认值 | 说明 |
|------|--------|------|
| **检测周期** | 3小时 | 每3小时执行一次健康检测 |
| **失败阈值** | 2次 | 连续2次检测失败才触发恢复操作 |
| **恢复超时** | 60秒 | 等待服务重启的最大时间 |
| **日志保留** | 48小时 | 健康日志保留时长 |

##### 2.5.2.3 恢复流程

```
开始检测
    │
    ▼
检测后端服务 ──失败──▶ 失败计数+1 ──▶ 连续2次失败？ ──是──▶ 重启后端
    │                              │                            │
    │                              否                            │
    ▼                              ▼                            ▼
检测前端服务 ──失败──▶ 失败计数+1 ──▶ 连续2次失败？ ──是──▶ 重启前端
    │                              │                            │
    │                              否                            │
    ▼                              ▼                            ▼
检测OpenClaw ──失败──▶ 失败计数+1 ──▶ 连续2次失败？ ──是──▶ 重启Gateway
    │                              │                            │
    │                              否                            │
    ▼                              ▼                            ▼
记录日志    ◀──────────── 结束
```

##### 2.5.2.4 日志记录

健康检测日志记录以下信息：
- 检测时间戳
- 服务名称
- 检测结果（健康/异常）
- 错误信息（如有）
- 执行的操作（无/重启/通知）

#### 2.5.3 数据模型

##### HealthLog 表

| 字段 | 类型 | 约束 | 描述 |
|------|------|------|------|
| `id` | `Integer` | `PRIMARY KEY, AUTOINCREMENT` | 主键 |

#### 2.5.8 外部进程守护（launchd）

##### 2.5.8.1 需求背景

内部健康检测调度器（HealthCheckScheduler）依赖于后端服务本身在运行。当后端服务完全停止时，调度器也随之停止，无法实现真正的自动恢复。因此需要引入外部独立的进程守护机制。

##### 2.5.8.2 方案设计

采用 macOS 原生 **launchd** 服务管理工具，实现系统级的进程监控和自动重启：

| 特性 | 说明 |
|------|------|
| **KeepAlive** | 服务崩溃时自动重启 |
| **RunAtLoad** | 系统启动时自动加载服务 |
| **ThrottleInterval** | 防止频繁重启（最小间隔10秒） |
| **日志管理** | 标准输出和错误输出分别记录 |

##### 2.5.8.4 启动脚本增强

`start.sh` 增加 `LAUNCHD_MODE` 环境变量支持：

```bash
# 在 launchd 模式下，保持主进程运行以便监控
if [ "$LAUNCHD_MODE" = "true" ]; then
    echo "🔄 启动 launchd 监控模式..."
    # 等待后端进程结束
    wait $BACKEND_PID
    EXIT_CODE=$?
    echo "⚠️  后端进程已退出，退出码: $EXIT_CODE"
    exit $EXIT_CODE
fi
```

当后端进程退出时，主脚本也会退出，launchd 检测到主进程退出后会自动重启整个服务。
