## 1. 项目概述

### 1.1 项目简介
**龙虾办公室**是一个 AI 团队工作空间监控系统，提供上帝视角的实时监控和干预能力。系统通过实时采集和分析 OpenClaw 平台的 Agent 状态、任务执行情况和大模型请求数据，为管理者提供全面的 AI 团队运行状况视图。

### 1.2 核心价值
- 📊 **实时监控**：实时查看 Agent 状态（休闲/对话/办公）和任务执行情况
- 📋 **任务管理**：任务清单、完成情况统计和任务移交功能
- 📈 **数据统计**：大模型请求数据统计、Token 使用统计和分模型分析
- 🔔 **督促功能**：及时干预 Agent 工作，确保任务按时完成
- 📤 **任务移交**：支持 Agent 间任务转交，优化工作分配

### 1.3 目标用户
- **AI 团队管理者**：需要全局监控团队运行状况
- **系统运维人员**：需要监控系统资源使用情况
- **Agent 开发者**：需要了解 Agent 性能和 Token 消耗

## 2. 功能需求 (Functional Requirements)

### 2.1 Agent 状态监控

| 功能点 | 用户故事 | 输入/输出 | 详细说明 |
|--------|----------|-----------|----------|
| **三区域状态显示** | 作为管理者，我希望看到 Agent 按状态分类显示，以便快速了解团队整体状态 | 输入：系统自动采集 Agent 状态<br>输出：三区域（办公区/对话区/休闲区）展示 | 1. 办公区（💼）：执行任务中<br>2. 对话区（💬）：与用户对话中<br>3. 休闲区（😴）：空闲待命 |
| **状态判定逻辑** | 作为系统，我需要根据 Agent 行为准确判定其状态 | 输入：Agent 活动数据<br>输出：状态判定结果 | 优先级：subagent 执行中 > cron 任务执行中 > 长时间无响应 > 任务上下文 + 最近活跃 > 工具调用频繁 + 最近活跃 > 对话活跃 > 默认 |
| **实时状态更新** | 作为管理者，我希望看到 Agent 状态的实时变化，以便及时了解团队动态 | 输入：WebSocket 事件<br>输出：状态实时更新 | 通过 WebSocket 实现状态实时推送，无需手动刷新 |

### 2.2 任务管理

| 功能点 | 用户故事 | 输入/输出 | 详细说明 |
|--------|----------|-----------|----------|
| **今日任务清单** | 作为管理者，我希望看到每个 Agent 的今日任务清单，以便了解工作安排 | 输入：任务数据<br>输出：按 Agent 分组的任务列表 | 显示今日创建或应执行的任务，包含任务状态和进度 |
| **任务过滤** | 作为管理者，我希望系统智能过滤任务，只显示相关任务 | 输入：时间、任务类型<br>输出：过滤后的任务列表 | 1. 不显示每周 X 的任务（除非今日是周 X）<br>2. 不显示每月 X 日的任务（除非今日是 X 日）<br>3. 不显示 1 小时前已过的每日任务 |
| **任务移交** | 作为管理者，我希望能在 Agent 间移交任务，以便优化工作分配 | 输入：移交信息（来源、目标、任务、说明）<br>输出：移交结果 | 支持一次性任务移交，自动更新任务归属并通知接收方 |

### 2.3 数据统计

| 功能点 | 用户故事 | 输入/输出 | 详细说明 |
|--------|----------|-----------|----------|
| **大模型请求统计** | 作为管理者，我希望看到大模型请求的详细统计，以便了解系统使用情况 | 输入：请求日志数据<br>输出：统计图表和数据 | 包含每小时请求量、服务商分布、Token 消耗等 |
| **分模型统计** | 作为开发者，我希望看到不同模型的使用情况，以便优化模型选择 | 输入：模型请求数据<br>输出：按模型分组的统计数据 | 显示每个模型的请求次数、Token 消耗等 |
| **Token 使用统计** | 作为管理者，我希望看到 Token 使用情况和预算执行情况，以便控制成本 | 输入：Token 消耗数据<br>输出：使用统计和预算分析 | 包含今日总消耗、日预算、使用率等指标 |

### 2.4 督促功能

| 功能点 | 用户故事 | 输入/输出 | 详细说明 |
|--------|----------|-----------|----------|
| **自动督促** | 作为系统，我需要在特定条件下自动督促 Agent，确保任务按时完成 | 输入：Agent 状态和任务数据<br>输出：督促通知 | 触发条件：任务超时、长时间无更新、Token 超预算、进度停滞 |
| **手动督促** | 作为管理者，我希望能手动发送督促消息给指定 Agent | 输入：Agent ID、督促内容<br>输出：督促通知 | 支持自定义督促消息，通过飞书和 WebSocket 发送 |

## 4. 数据需求

### 4.1 数据采集（从 OpenClaw 获取数据）

| 数据源 | 采集方式 | 采集频率 | 存储位置 |
|--------|----------|----------|----------|
| **Agent 状态** | 从 OpenClaw 会话文件分析 | 每 5 分钟 | AgentStatus 表 |
| **任务数据** | 从 OpenClaw 会话文件分析 | 每 10 分钟 | TaskRecord 表 |
| **请求日志** | 从 OpenClaw 会话文件分析 | 每 15 分钟 | RequestLog 表 |
| **Cron 任务** | 从配置文件同步 | 每 90 分钟 | TaskRecord 表 |

**说明**：系统通过定时任务自动从 OpenClaw 平台采集数据，确保数据的实时性和准确性。

### 4.2 页面同步频率

| 数据类型 | 同步方式 | 同步频率 | 说明 |
|----------|----------|----------|------|
| **页面自动刷新** | 前端定时请求 | 可配置（默认 300 秒） | 自动刷新所有数据 |
| **手动刷新** | 用户点击刷新按钮 | 按需触发 | 立即刷新所有数据 |
| **WebSocket 推送** | 后端实时推送 | 实时 | Agent 状态变化时推送 |
| **状态数据** | API 调用 | 可配置（默认 300 秒） | Agent 状态、任务状态 |
| **统计数据** | API 调用 | 可配置（默认 300 秒） | 请求统计、Token 统计 |

**说明**：系统采用多种同步机制确保数据的实时性和准确性，同时平衡服务器负载。

### 4.3 数据模型

| 数据模型 | 主要字段 | 用途 |
|----------|----------|------|
| **AgentStatus** | agent_id, status, task_id, task_name, progress, token_used, last_activity | 存储 Agent 实时状态 |
| **TaskRecord** | task_id, task_name, agent_id, status, priority, started_at, completed_at, token_used | 存储任务执行记录 |
| **RequestLog** | request_id, agent_id, provider, model_name, tokens_total, created_at | 存储大模型请求日志 |
| **TokenLog** | agent_id, task_id, token_count, token_type, action, created_at | 存储 Token 使用明细 |
| **ReminderLog** | agent_id, task_id, reminder_type, response_status, created_at | 存储督促记录 |
| **TaskHandover** | handover_id, task_id, from_agent_id, to_agent_id, status, created_at | 存储任务移交记录 |

### 4.4 配置文件

**配置文件路径**：`frontend/config.js`

**配置文件格式**：JavaScript 对象（可转换为 JSON）

**配置示例**：

```javascript
// 龙虾办公室配置文件
// 刷新频率设置（单位：秒）
window.LOBSTER_CONFIG = {
  // API 基础配置
  API_BASE: 'http://localhost:8000',
  BACKEND_PORT: 8000,
  
  // WebSocket 配置
  WS_URL: 'ws://localhost:8000/ws',
  
  // 刷新频率设置（单位：秒）
  refresh: {
    // 页面自动刷新频率（30-600秒）
    auto_refresh: 300,
    
    // 状态数据刷新频率（30-600秒）
    status_data: 300,
    
    // 统计数据刷新频率（30-600秒）
    stats_data: 300,
    
    // 后端同步频率（单位：分钟）
    agent_sync: 5,
    task_sync: 10,
    request_sync: 15,
    cron_sync: 90
  },
  
  // Token 预算配置
  token_budget: {
    daily: 500000,
    monthly: 15000000
  }
};
```

**配置验证规则**：
- 前端刷新频率（单位：秒）：30-600 秒之间
- 后端同步频率（单位：分钟）：1-120 分钟之间
- 若配置无效或未提供，系统将使用默认值

**配置加载流程**：
1. 系统首先尝试加载 `frontend/config.js` 配置文件
2. 对配置进行验证，确保刷新频率在有效范围内
3. 若配置有效，使用配置值；若无效或未提供，使用默认值
4. 控制台会输出当前使用的刷新频率设置

### 4.5 配置界面

**界面位置**：使用统计面板下方左侧，紫色配置按钮

**功能说明**：
- 提供图形化界面修改系统配置
- 支持修改前端刷新频率和后端同步频率
- 实时验证输入值的有效性
- 保存配置后自动生效

**配置项**：

| 配置项 | 单位 | 范围 | 默认值 | 说明 |
|--------|------|------|--------|------|
| 页面自动刷新 | 秒 | 30-600 | 300 | 前端页面自动刷新频率 |
| 状态数据刷新 | 秒 | 30-600 | 300 | 状态数据API调用频率 |
| 统计数据刷新 | 秒 | 30-600 | 300 | 统计数据API调用频率 |
| Agent 状态同步 | 分钟 | 1-120 | 5 | 从OpenClaw同步Agent状态的频率 |
| 任务数据同步 | 分钟 | 1-120 | 10 | 从OpenClaw同步任务数据的频率 |
| 请求日志同步 | 分钟 | 1-120 | 15 | 从OpenClaw同步请求日志的频率 |
| Cron 任务同步 | 分钟 | 1-120 | 90 | 从OpenClaw同步Cron任务的频率 |

**配置保存机制**：
1. 前端配置保存到浏览器 localStorage
2. 后端配置通过 API 发送到服务器
3. 服务器保存配置到配置文件并重启调度器
4. 页面自动刷新以应用新配置
